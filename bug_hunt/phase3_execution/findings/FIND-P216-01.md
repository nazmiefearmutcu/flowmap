### Repro
1. `_gui_tick` sets `self._order_book.on_trade = None` before apply.
2. Restores with `self._order_book.on_trade = self._on_trade` only if apply block completes.
3. Force exception in `apply_snapshot` / `apply_updates` / `record_trades` (malformed msg, engine bug).
4. Exception aborts `_gui_tick`; restore never runs → `on_trade` stays `None`.
5. Subsequent non-batch callers of `record_trade` / signal path lose heatmap/pulse/VP trade UI.

### Expected
Callback always restored (or better: `record_trades(..., notify=False)` without mutating attribute).

### Actual
No `try/finally`. One fault permanently disables per-trade UI callbacks until restart or rewire.

### Fix hint
```python
prev = self._order_book.on_trade
try:
    self._order_book.on_trade = None
    ...
finally:
    self._order_book.on_trade = prev
```
Or `record_trades(trades, notify=False)` API on OrderBook.

### Evidence
```934:947:flowmap/ui/main_window.py
self._order_book.on_trade = None
...
self._order_book.on_trade = self._on_trade  # not in finally
```

### Fix (2026-07-13)
Phase-4: `prev_on_trade` + `try/finally` restore in `_gui_tick`. See `bug_hunt/phase4_fixes/FIX_TICK_QUEUE.md`.
