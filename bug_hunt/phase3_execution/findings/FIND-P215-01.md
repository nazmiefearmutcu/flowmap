### Repro
1. Drain batch order: trade T1 → snapshot S → trade T2.
2. Code path:
   - On snapshot: `updates.clear()`, `bbos.clear()` — **trades list untouched**.
   - Apply: `apply_snapshot(S)` then `record_trades([T1, T2])`.
3. T1 is applied **after** S even though exchange time order was T1 then S (S already reflects post-T1 book).

### Expected
Either chronological per-message apply, or clear/partition trades pre-snapshot like updates; document explicit policy.

### Actual
`record_trades` after snapshot can re-absorb size already reflected in S → punched levels / wrong residual size vs lake truth.

### Fix hint
Preferred: single FIFO apply loop per drained message. If keep batching: drop trades collected before last snapshot in the batch (or clear trades on snapshot).

### Evidence
```914:944:flowmap/ui/main_window.py
if msg_type == "snapshot":
    snapshots.append(obj)
    updates.clear()
    bbos.clear()
    # trades NOT cleared
...
if trades:
    self._order_book.record_trades(trades)
```

### Fix (2026-07-13)
Phase-4: `trades.clear()` on snapshot in drain batch. Policy: only trades after last snap in the batch are applied. See `bug_hunt/phase4_fixes/FIX_TICK_QUEUE.md`.
