### Repro
1. Inspect `_cryp_book_delta_to_flowmap_updates`: each bid/ask level becomes a separate `Level2Update`.
2. `_dispatch_record` for non-snapshot `book_delta` returns that list.
3. Emit loop (`crypcodile_replay.py:511-517`) puts **one queue item per update object**.
4. Same fan-out exists on live path via shared `_dispatch_record` pattern in live sink.

### Expected
One book event ≈ O(1) queue messages (or conflated book state), not O(depth).

### Actual
A single delta with D levels enqueues D messages. Dense lakes at speed=0 easily exceed drain μ (≈62.5k msgs/s theoretical), driving FIND-P213-01 growth and FIND-P214 lag.

### Fix hint
Put one composite update list per delta, or conflate latest book on worker side (~33Hz like `crypto._sender_loop`); apply as batch in OrderBook without per-level queue hops.
