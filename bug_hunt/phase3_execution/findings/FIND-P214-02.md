### Repro
1. Enqueue 5000 `update` messages, then 1 `snapshot`, then more updates (synthetic queue).
2. Call `_gui_tick` repeatedly with `limit=1000`.
3. Snapshot is applied only after 5 full ticks of updates (≈80ms+ structural, worse under load).
4. Within-batch `updates.clear()` on snapshot (P2-15) **never runs** for that snapshot until it enters the same drain batch.

### Expected
Latest snapshot should be prioritised or scanned so full-replace book state is not delayed many ticks behind obsolete deltas.

### Actual
Strict FIFO + hard cap: a later snapshot waits behind every preceding message. Cross-tick, pre-snapshot updates already applied remain until a later snapshot arrives.

### Fix hint
Two-phase drain: scan batch/backlog for latest snapshot index; drop pre-snapshot updates; or dual-queue (book state vs trades). Budget-based drain alone does not fix priority.

### Evidence
- FIND-P214-01 hard cap at `:908`
- No snapshot priority / conflation in drain loop
